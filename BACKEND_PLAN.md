# Backend Plan — Task App

## Overview

Real-time multi-user task management app with organizations (groups for family/teams), subtasks, recurring tasks, time tracking, milestones, comments with @mentions, and real-time websocket updates.

## Tech Stack

| Layer | Tech |
|---|---|
| Server | Fastify 5 |
| ORM | Drizzle ORM 0.45 + drizzle-kit |
| DB Driver | postgres.js (postgres) |
| Auth | better-auth (email/password + Google OAuth + organization plugin) |
| DB | PostgreSQL (Neon/Supabase) |
| Real-time | ws via @fastify/websocket |
| Validation | Zod (shared `@meu-projeto/types`) |

## Middleware Chain

```
Request
  -> @fastify/cors (credentials: true, origin: CORS_ORIGIN)
  -> auth.middleware (validates session cookie, decorates request.user)
  -> org.middleware (validates :groupId membership, decorates request.member with role)
  -> controller (calls service)
  -> service (business logic, agnostic of protocol)
  -> db (drizzle)
```

### auth.middleware.ts
- Reads session cookie/token from request
- Calls `auth.api.getSession()` to validate
- Decorates `request.user` with session user
- Returns 401 if no session

### org.middleware.ts
- Reads `:groupId` param
- Checks `member` table for `userId + organizationId`
- Decorates `request.member` with role (owner|admin|member)
- Returns 403 if not a member

## WebSocket

### Connection
- Route: `GET /ws?token=<session_token>`
- Browser can't set custom headers on WebSocket, so auth via query param
- On connect: validate token via `auth.api.getSession()`, store `{ socket, userId, organizationIds[] }` in a `Map`

### Broadcast helpers (ws.service.ts)
- `broadcastToOrg(orgId, event, data)` — iterates connected clients, sends to those in the org
- `broadcastToUser(userId, event, data)` — send to a single user (for notifications/mentions)

### Events
| Event | Payload | Recipients |
|---|---|---|
| `task:created` | task object | all org members |
| `task:updated` | task object | all org members |
| `task:deleted` | { taskId } | all org members |
| `comment:created` | comment object | all org members |
| `notification:new` | notification object | specific user only |

### Client tracking
- `Map<WebSocket, { userId: string, organizationIds: string[] }>`
- On `socket.on('close')`: remove from map
- On org switch: client sends `{ event: 'switch_org', orgId }` to update tracking

## Recurring Task Logic (recurrence.service.ts)

When `task.service.ts` lists tasks for an org, before returning:

1. Query all `recurrence_rules` where `nextDueAt <= now()`.
2. For each overdue rule:
   a. Create a new `task` instance (clone of the template task):
      - New id
      - `dueDate = nextDueAt + completionWindowMinutes`
      - `status = 'todo'`
      - `parentId = null` (instance, not subtask)
      - Copy: title, description, priority, assigneeId, organizationId, timeEstimateMinutes
   b. Update `recurrence_rules.lastGeneratedAt = now()`
   c. Calculate next `nextDueAt` based on frequency + interval
   d. If `nextDueAt > endDate` (or no endDate), keep generating until caught up
3. Include new instances in the returned list.

This is done **synchronously at read time** — no background worker needed.

Example: User says "I need this task available every 7 days, with 12h to complete"
- `frequency = 'weekly'`, `interval = 7`, `completionWindowMinutes = 720`
- `nextDueAt` = first occurrence date
- When `nextDueAt` passes, a new task instance is created with `dueDate = nextDueAt + 720min`

## Time Tracking

- **Manual log entries**: Users log time entries (startedAt, endedAt, durationMinutes, note)
- **Remaining time calculation**: `timeEstimateMinutes - SUM(time_entries.durationMinutes) WHERE taskId = X`
- **UI display**: "1 hour remaining", "1 day and 12 hours remaining" (calculated on frontend from the estimate and total logged)
- **Stats endpoint**: Aggregated time per user, per task, per org

## Implementation Order

1. **Env setup** — `env.ts`, `.env`, `.env.example`
2. **DB layer** — `drizzle.config.ts`, `db/schema/*`, `db/relations.ts`, `db/client.ts`
3. **Auth** — `auth/auth.ts` (better-auth + drizzle adapter + org plugin + google), `auth/session.ts`
4. **Middlewares** — `auth.middleware.ts`, `org.middleware.ts`
5. **WebSocket** — `ws/ws.ts`, `services/ws.service.ts`
6. **Services** — implement each service (pure business logic, no Fastify dependency)
7. **Controllers** — thin wrappers calling services
8. **Routes** — Fastify route definitions, zod validation, middleware chain
9. **Index** — `index.ts` boot file, register everything
10. **Migrations** — `drizzle-kit generate` + `migrate`

## Key Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Groups model | better-auth organization plugin | Built-in tables, roles, invitations — no custom code |
| Recurring tasks | On-demand generation at read time | No background worker/cron needed |
| Time tracking | Manual log entries | Simpler, no live timer state to manage via ws |
| Mentions | Parse @username from comment content | Create notification + ws broadcast to mentioned user |
| Subtasks | Self-referencing parentId on tasks | Unlimited nesting, simple queries |
| Tags per org | Scoped to organization | Reusable across tasks within a group |
| Auth on ws | Query param token | Browser limitation — can't set custom headers on WebSocket |
